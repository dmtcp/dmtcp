# Internal Plugin Consolidation: Staged Migration Plan

Reader: a DMTCP maintainer or future implementation agent preparing the M003 consolidation edits. Post-read action: execute or review each stage gate for timer, svipc, ipc, pid, alloc, and dl without re-planning the source, build, launcher, exec propagation, rollback, stale-artifact, duplicate-symbol, disable-flag, and focused-test contracts.

This file is the migration appendix for folding selected internal plugin behavior into `libdmtcp.so`. It starts with a current-source stage map, then defines the implementation gates that future M003/M004 work must satisfy one stage at a time before removing an old internal DSO boundary.

## S01 preflight first gate

Before the timer stage or any later source movement, the mandatory first review gate is the S01 preflight in `doc/internal-plugin-consolidation-preflight.md`. Reviewers should run `python3 test/verify-consolidation-preflight.py integration` and then `python3 test/verify-consolidation-preflight.py full` before timer, SysV IPC, IPC, PID, alloc, or DL source movement to confirm the enable-state model, wrapper ownership, duplicate-symbol command classes, stale-artifact cleanup, rollback gates, and external ABI boundary are named.

This gate is still a source-backed contract, not runtime proof. It blocks starting a stage when the preflight is missing or uncross-linked, but clean rebuilds, checkpoint/restart behavior, sanitizer behavior, `RTLD_NEXT`, `dl_iterate_phdr`, and external plugin compatibility evidence remain M003, M004, or M005 handoffs.

## S02 registration gate

After the S01 preflight and before timer, SysV IPC, IPC, PID, alloc, or DL source movement, the S02 registration gate is the mandatory appendix integration guard: review `doc/internal-plugin-consolidation-registration-architecture.md` and run `python3 test/verify-consolidation-registration-plan.py integration` plus `python3 test/verify-consolidation-registration-plan.py full`. This gate keeps the public external ABI unchanged through `DMTCP_DECL_PLUGIN`, routes former descriptor-bearing built-ins through a PluginManager-owned built-in descriptor table with non-public accessors, enforces no duplicate internal initializer definitions, preserves alloc/dl wrapper-only behavior and pid-last placement, and records no runtime proof claims.

## Accepted staged migration order

The accepted migration order is timer, svipc, ipc, pid, alloc, dl. Written compactly for static checks: timer,svipc,ipc,pid,alloc,dl. This follows D002's risk ladder: start with the smallest descriptor-based tracer bullet, then move the bounded SysV IPC plugin, then migrate the broader IPC registration surface, then preserve pid last, and only then move wrapper-only alloc and DL behavior. ## Current launcher and exec propagation surfaces

`src/dmtcp_launch.cpp` is the launch-time authority for the generated DMTCP chain. It appends enabled built-in libraries, records that generated chain in `ENV_VAR_HIJACK_LIBS`, and handles `--disable-all-plugins` by reducing the generated chain to `libdmtcp.so`. It also reads `ENV_VAR_ALLOC_PLUGIN` and `ENV_VAR_DL_PLUGIN` to decide whether `libdmtcp_alloc.so` and `libdmtcp_dl.so` enter the generated chain. [src/dmtcp_launch.cpp]

`src/util_exec.cpp` keeps the restart command-line surface aligned with those launch settings. `Util::getDmtcpArgs` observes `ENV_VAR_ALLOC_PLUGIN`, `ENV_VAR_DL_PLUGIN`, and plugin arguments, then emits `--disable-alloc-plugin`, `--disable-dl-plugin`, and `--with-plugin` as needed. This surface is the reason alloc and DL cannot be treated as simple build deletions; their disable flags remain user-visible behavior after consolidation. [src/util_exec.cpp]

`src/execwrappers.cpp` carries the generated chain across exec. `getUpdatedLdPreload` reads `ENV_VAR_HIJACK_LIBS`, the 32-bit companion `ENV_VAR_HIJACK_LIBS_M32`, and `ENV_VAR_ORIG_LD_PRELOAD` while rebuilding the preload environment for the new program image. A stage that removes an internal DSO from the generated chain must keep this exec propagation behavior coherent with launch-time capture. [src/execwrappers.cpp]

## Current PluginManager and public ABI boundary

`PluginManager::initialize` creates the singleton and enters the registration chain by calling the visible `dmtcp_initialize_plugin`. The C entry `dmtcp_register_plugin` appends descriptors to `pluginInfos`, and `PluginManager::eventHook` then delivers ordinary events in registration order while delivering resume and restart classes in reverse order. [src/pluginmanager.cpp] [src/pluginmanager.h]

The public ABI remains in `include/dmtcp.h`: external plugins use the weak `dmtcp_initialize_plugin`, `DMTCP_DECL_PLUGIN`, `NEXT_FNC(dmtcp_initialize_plugin)`, and `DmtcpPluginDescriptor_t`. The migration plan must keep that ABI stable for external shared-object plugins while removing duplicate internal initializer definitions from consolidated built-ins. [include/dmtcp.h] [src/pluginmanager.cpp]

The current `libdmtcp.so` core descriptor order is path translator, syslog, rlimit-float, alarm, terminal, coordinator, processinfo, uniquepid. That order is preserved inside the future table between the pre-core internal plugin stages and the pid-last stage unless a later implementation task produces source-backed runtime evidence for changing it. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h]

## Current IPC subplugin and pid-last ordering

The IPC shared object has a handwritten initializer in `src/plugin/ipc/ipc.cpp`, not a direct `DMTCP_DECL_PLUGIN` use. Its forward registration order is `ssh,event,file,pty,socket`: SSH from `src/plugin/ipc/ssh/ssh.cpp`, event from `src/plugin/ipc/event/eventconnlist.cpp`, file from `src/plugin/ipc/file/fileconnlist.cpp`, pty from `src/plugin/ipc/file/ptyconnlist.cpp`, and socket from `src/plugin/ipc/socket/socketconnlist.cpp`. [src/plugin/ipc/ipc.cpp] [src/plugin/ipc/ssh/ssh.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp]

Because PluginManager restart and resume traversal is reverse-order, that forward sequence intentionally yields `socket,pty,file,event,ssh` during reverse restore-style events. The IPC stage is therefore a registration-order migration, not only a build-boundary migration. [src/pluginmanager.cpp] [src/plugin/ipc/ipc.cpp] [src/plugin/ipc/ssh/ssh.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp]

PID is separate from IPC and remains pid last. `src/dmtcp_launch.cpp` places `libdmtcp_pid.so` after `libdmtcp.so`, and `src/plugin/pid/pid.cpp` declares the PID descriptor through the public macro. The PID stage must keep that pid last property when the separate `libdmtcp_pid.so` artifact is removed. [src/dmtcp_launch.cpp] [src/plugin/pid/pid.cpp]

## Current alloc and dl wrapper-only boundaries

Alloc and DL are currently wrapper-only groups. `src/plugin/alloc/mallocwrappers.cpp` and `src/plugin/alloc/mmapwrappers.cpp` provide allocator and mapping wrappers, including `dmtcp_alloc_enabled`, `malloc`, `free`, `mmap`, and `munmap`; neither file should gain a descriptor row. The migration invariant is no descriptor accessor and no `DmtcpPluginDescriptor_t` for alloc. [src/plugin/alloc/mallocwrappers.cpp] [src/plugin/alloc/mmapwrappers.cpp] [src/dmtcp_launch.cpp]

`src/plugin/dl/dlwrappers.cpp` provides the `dlopen` and `dlclose` wrapper layer, tracks `thread_performing_dlopen_dlsym`, and uses `WRAPPER_EXECUTION_DISABLE_CKPT` around sensitive dl activity. DL also remains wrapper-only: no descriptor accessor, no `DmtcpPluginDescriptor_t`, and no event-hook row in the future built-in table. [src/plugin/dl/dlwrappers.cpp] [src/dmtcp_launch.cpp]

The user-facing enable surface for these wrapper-only groups crosses launch and restart propagation. `src/dmtcp_launch.cpp` reads `ENV_VAR_ALLOC_PLUGIN` and `ENV_VAR_DL_PLUGIN`; `src/util_exec.cpp` maps those variables back to `--disable-alloc-plugin` and `--disable-dl-plugin` through `Util::getDmtcpArgs`. Consolidation must therefore keep wrapper-only disable behavior observable even when separate `libdmtcp_alloc.so` and `libdmtcp_dl.so` artifacts are gone. [src/dmtcp_launch.cpp] [src/util_exec.cpp]

## Current build and focused test boundaries

The build boundary is split today. `src/Makefile.am` builds `libdmtcp.so` and already contains sources such as `src/pluginmanager.cpp`, `src/execwrappers.cpp`, and `src/util_exec.cpp`. `src/plugin/Makefile.am` builds the separate internal plugin DSOs for alloc, DL, IPC, PID, SysV IPC, and timer. The checkout uses this one aggregate plugin makefile; there are no per-stage plugin makefiles to edit. [src/Makefile.am] [src/plugin/Makefile.am]

Future M003 implementation should move one stage at a time by adding the stage source files to the `libdmtcp.so` boundary in `src/Makefile.am`, removing only that old DSO from `src/plugin/Makefile.am`, and keeping launch, exec, and disable behavior aligned. That staged build movement is what makes rollback practical: the previous stage can still be compared against the current source map if the next stage fails. [src/Makefile.am] [src/plugin/Makefile.am] [src/dmtcp_launch.cpp] [src/execwrappers.cpp]

Focused tests come from `test/autotest.py` and the test source files it names. The timer stage names `timer1` and `clock` as required coverage, keeps `timer2` optional for environments where the existing commented test is enabled, names `sysv-shm1` and `sysv-shm2` for SysV shared memory including `IPC_PRIVATE`, names `sysv-sem` and `sysv-msg` for related SysV IPC coverage, names `dlopen1` and `dlopen2` for the DL wrapper surface, and uses `plugin-init` as an initialization smoke source. The timer source boundary for that first stage is not only `src/plugin/timer/timerlist.cpp` and `src/plugin/timer/timerwrappers.cpp`: it also includes `src/plugin/timer/timer_create.cpp`, `src/plugin/timer/timerlist.h`, and `src/plugin/timer/timerwrappers.h` because `src/plugin/timer/timerwrappers.cpp` calls `timer_create_sigev_thread()` from `src/plugin/timer/timer_create.cpp`.
