# Internal Plugin Consolidation: Registration Architecture Map

Reader: a DMTCP maintainer preparing the S02 registration architecture for internal-plugin consolidation. Post-read action: identify today's descriptor registration chain, wrapper-only groups, build boundaries, and order constraints before designing a PluginManager-owned built-in descriptor table.

This appendix is intentionally limited to the current 2.5 source map and the S01 risk constraints that S02 must preserve. Later design sections can add a proposed table and migration plan, but this current map is the baseline that must remain locally checkable by `python3 test/verify-consolidation-registration-plan.py current-map`. [test/verify-consolidation-registration-plan.py]

## Scope and evidence boundary

The current registration surface is defined by `PluginManager::initialize`, `dmtcp_register_plugin`, the weak `dmtcp_initialize_plugin` entry point, and the public `DMTCP_DECL_PLUGIN` macro. `PluginManager::initialize` creates the singleton and calls the first visible `dmtcp_initialize_plugin`; `dmtcp_register_plugin` appends each received descriptor to the manager-owned vector. [src/pluginmanager.cpp] [include/dmtcp.h]

The scope here is current behavior only: launch-time preload construction, shared-object build boundaries, descriptor declarations, IPC subplugin order, and the wrapper-only status of alloc and dl. The S01 constraint carried forward is that consolidation must not be treated as only a link edit; it must preserve dynamic-loader ordering, plugin registration order, forward and reverse event delivery, and the distinction between reducing DMTCP-owned preload layers and proving any sanitizer outcome. [doc/internal-plugin-consolidation-wrapper-order-tsan.md] [src/pluginmanager.cpp] [include/dmtcp.h]

## Current dmtcp_launch preload and build boundary

`dmtcp_launch` constructs the DMTCP-generated preload string in `setLDPreloadLibs`. If `DMTCP_PLUGIN` is present, its value is prepended before the DMTCP built-in table. The enabled default `pluginInfo[]` entries then add `libdmtcp_alloc.so`, `libdmtcp_dl.so`, `libdmtcp_ipc.so`, `libdmtcp_svipc.so`, `libdmtcp_timer.so`, `libdmtcp.so`, and `libdmtcp_pid.so`; disabled-by-default entries such as path virtualization sit in the same table but are not part of the ordinary enabled chain. `libdmtcp_pid.so` is explicitly annotated as the plugin that must come last. [src/dmtcp_launch.cpp]

`ENV_VAR_HIJACK_LIBS` is set after the generated chain is assembled and before any original user `LD_PRELOAD` is appended to the final process preload value. That means the architecture map must distinguish the generated DMTCP chain from the final environment value passed to the target. [src/dmtcp_launch.cpp]

The build boundary matches the preload boundary. `src/Makefile.am` builds `libdmtcp.so` with `src/pluginmanager.cpp`, the core descriptor sources, and the base wrappers, while `src/plugin/Makefile.am` builds `libdmtcp_alloc.so`, `libdmtcp_dl.so`, `libdmtcp_ipc.so`, `libdmtcp_svipc.so`, `libdmtcp_timer.so`, and `libdmtcp_pid.so` as separate plugin DSOs. [src/Makefile.am] [src/plugin/Makefile.am]

## Current PluginManager registration chain

`PluginManager::initialize` creates the singleton only once, then calls `dmtcp_initialize_plugin` if that weak symbol is available. The public C function `dmtcp_register_plugin` asserts that the singleton exists and delegates to `PluginManager::registerPlugin`, which wraps the descriptor in `PluginInfo` and appends it to `pluginInfos`. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h]

The libdmtcp implementation of `dmtcp_initialize_plugin` registers core descriptors in this order: path translator, syslog, rlimit-float, alarm, terminal, coordinator API, process info, and unique pid. It then calls `NEXT_FNC(dmtcp_initialize_plugin)`, so anything later in the dynamic-linker chain, especially `libdmtcp_pid.so`, can still register after the core libdmtcp descriptors. [src/pluginmanager.cpp] [include/dmtcp.h]

`PluginManager::eventHook` depends on the resulting vector order. It delivers pre-checkpoint-style events in registration order and delivers `DMTCP_EVENT_RESUME`, `DMTCP_EVENT_RESTART`, and related unwind events in reverse order. That reverse order behavior is not incidental; the source comment names it as required for layered software. [src/pluginmanager.cpp] [src/pluginmanager.h]

## Current external plugin ABI and DMTCP_DECL_PLUGIN chain

The public ABI remains the weak `dmtcp_initialize_plugin` symbol plus `DMTCP_DECL_PLUGIN`. The macro expands to a `dmtcp_initialize_plugin` definition that calls `dmtcp_register_plugin(descr)`, looks up `NEXT_FNC(dmtcp_initialize_plugin)`, and calls that next function when present. External plugins that use `DMTCP_DECL_PLUGIN` therefore participate in the same `dmtcp_initialize_plugin` chain as current built-ins. [include/dmtcp.h] [src/pluginmanager.cpp]

Current built-ins use that ABI in two ways. Timer, SysV IPC, and pid each declare a `DmtcpPluginDescriptor_t` and use `DMTCP_DECL_PLUGIN` directly. IPC uses a handwritten `dmtcp_initialize_plugin` because it first registers multiple IPC subplugin descriptors, then calls `NEXT_FNC(dmtcp_initialize_plugin)`. [src/plugin/timer/timerlist.cpp] [src/plugin/svipc/sysvipc.cpp] [src/plugin/pid/pid.cpp] [src/plugin/ipc/ipc.cpp] [include/dmtcp.h]

The external ABI boundary for consolidation is therefore strict: keep external `DMTCP_DECL_PLUGIN` users on the existing `dmtcp_initialize_plugin` and `NEXT_FNC(dmtcp_initialize_plugin)` chain, while any future built-in table must avoid changing what external plugins see. [include/dmtcp.h] [src/pluginmanager.cpp]

## Current built-in and external ordering

The order that S02 architecture must preserve is a registration order, not merely a link list. Because `dmtcp_launch` prepends `DMTCP_PLUGIN` before DMTCP's built-in DSOs, an external `DMTCP_PLUGIN` chain that defines `dmtcp_initialize_plugin` registers before DMTCP internals. [src/dmtcp_launch.cpp] [include/dmtcp.h]

After the external chain, the current default DMTCP descriptor sequence is: IPC subplugins `ssh,event,file,pty,socket`, then svipc, then timer, then libdmtcp core descriptors, then pid last. The libdmtcp core descriptors are path translator, syslog, rlimit-float, alarm, terminal, coordinator, processinfo, and uniquepid. The separate pid descriptor is last because `libdmtcp_pid.so` follows `libdmtcp.so` in `pluginInfo[]` and uses the macro chain. [src/dmtcp_launch.cpp] [src/pluginmanager.cpp] [include/dmtcp.h]

This ordering also defines event behavior: forward events see external plugins first, then IPC, svipc, timer, libdmtcp core descriptors, and pid; `DMTCP_EVENT_RESUME` and `DMTCP_EVENT_RESTART` see the same vector in reverse order. A built-in descriptor table owned by PluginManager must reproduce that observable order unless a later design explicitly documents and validates a behavior change. [src/pluginmanager.cpp] [src/dmtcp_launch.cpp]

## Current IPC subplugin ordering

The IPC shared object does not use `DMTCP_DECL_PLUGIN`; its `dmtcp_initialize_plugin` explicitly calls `ipc_initialize_plugin_ssh`, `ipc_initialize_plugin_event`, `ipc_initialize_plugin_file`, `ipc_initialize_plugin_pty`, and `ipc_initialize_plugin_socket`, then chains with `NEXT_FNC(dmtcp_initialize_plugin)`. The compact order marker is `ssh,event,file,pty,socket`. [src/plugin/ipc/ipc.cpp]

The individual descriptors are `sshPlugin`, `eventPlugin`, `filePlugin`, `ptyPlugin`, and `socketPlugin`; each helper calls `dmtcp_register_plugin` with that descriptor. [src/plugin/ipc/ssh/ssh.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp]

The comments in IPC explain why forward order is chosen to support reverse restoration. File, pty, and socket are independent relative to each other, event restoration must happen after other file-descriptor plugins, and SSH restoration must happen after socket restoration. Since `PluginManager::eventHook` processes `DMTCP_EVENT_RESUME` and `DMTCP_EVENT_RESTART` in reverse order, forward registration `ssh,event,file,pty,socket` produces reverse order `socket,pty,file,event,ssh` during those restore events. [src/plugin/ipc/ipc.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp] [src/plugin/ipc/ssh/ssh.cpp]

## Current wrapper-only alloc and dl enable semantics

Alloc and dl are enabled through the launch engine, not through descriptor registration. `src/dmtcp_launch.cpp` reads `ENV_VAR_ALLOC_PLUGIN` and `ENV_VAR_DL_PLUGIN`, toggles `enableAllocPlugin` and `enableDlPlugin`, and includes or skips `libdmtcp_alloc.so` and `libdmtcp_dl.so` in `pluginInfo[]`. [src/dmtcp_launch.cpp]

`src/plugin/alloc/mallocwrappers.cpp` defines allocator wrappers such as `malloc`, `calloc`, `free`, and `realloc`, and brackets real allocator calls with `DMTCP_PLUGIN_DISABLE_CKPT` and `DMTCP_PLUGIN_ENABLE_CKPT`. It also exposes `dmtcp_alloc_enabled`, but it does not define `DmtcpPluginDescriptor_t` or `dmtcp_initialize_plugin`. [src/plugin/alloc/mallocwrappers.cpp]

`src/plugin/dl/dlwrappers.cpp` defines libdl wrappers around `dlopen` and `dlclose`, uses `NEXT_FNC(dlopen)` and `NEXT_FNC(dlclose)` for the real functions, and documents the `thread_performing_dlopen_dlsym` and `WRAPPER_EXECUTION_DISABLE_CKPT` locking/fast-pass reason for dl activity. It also has no `DmtcpPluginDescriptor_t` registration. [src/plugin/dl/dlwrappers.cpp]

The consolidation implication is that alloc/dl need an engine-controlled wrapper-only enable path, keyed to `ENV_VAR_ALLOC_PLUGIN` and `ENV_VAR_DL_PLUGIN`, rather than a descriptor accessor. Their future path can preserve wrapper enable/fast-pass semantics without pretending that alloc or dl currently contribute a `DmtcpPluginDescriptor_t` to `PluginManager`. [src/dmtcp_launch.cpp] [src/plugin/alloc/mallocwrappers.cpp] [src/plugin/dl/dlwrappers.cpp]

## Proposed PluginManager-owned built-in registration table

Recommended design D008: keep `libdmtcp.so`'s exported `dmtcp_initialize_plugin` as the external ABI chain node and make that function delegate to a PluginManager-owned built-in table at the same observable position where the libdmtcp chain node currently registers its core descriptors. Do not move the built-in registrations into `PluginManager::initialize` ahead of the `external dmtcp_initialize_plugin chain`; `PluginManager::initialize` must still create the singleton and enter the chain through `dmtcp_initialize_plugin`, so a user `DMTCP_PLUGIN` entry that appears before libdmtcp continues to register before DMTCP internals through `NEXT_FNC(dmtcp_initialize_plugin)` and `dmtcp_register_plugin`. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/dmtcp_launch.cpp]

The libdmtcp chain node should call a small internal helper, for example `PluginManager::registerBuiltInPlugins()` or a private file-local `registerBuiltInPluginTable()`, that loops over a fixed `built-in descriptor table`. The table order is the contract: pre-core IPC descriptors `ssh,event,file,pty,socket`, then `sysvipc`, then `timer`, then the existing core descriptors in the current `src/pluginmanager.cpp` order, and finally the post-core `pid` descriptor last. The resulting default forward order is external plugins first, then SSH, event, file, pty, socket, SysV IPC, timer, path translator, syslog, rlimit-float, alarm, terminal, coordinator API, process info, unique pid, and pid. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/dmtcp_launch.cpp]

The existing core portion stays byte-for-byte equivalent in ordering terms: `dmtcp_PathTranslator_PluginDescr()`, `dmtcp_Syslog_PluginDescr()`, `dmtcp_Rlimit_Float_PluginDescr()`, `dmtcp_Alarm_PluginDescr()`, `dmtcp_Terminal_PluginDescr()`, `CoordinatorAPI::pluginDescr()`, `dmtcp_ProcessInfo_PluginDescr()`, and `UniquePid::pluginDescr()`. Consolidation only changes how former separate built-in DSOs reach `PluginManager`; it must not change `PluginManager::eventHook` forward traversal or reverse traversal for `DMTCP_EVENT_RESUME` and `DMTCP_EVENT_RESTART`. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/dmtcp_launch.cpp]

## Descriptor accessor naming and ABI boundary

Use the existing by-value `descriptor accessor` convention already visible in libdmtcp: functions named like `dmtcp_*_PluginDescr()` return `DmtcpPluginDescriptor_t` by value, while class-owned descriptors can use the `UniquePid::pluginDescr()` style. New consolidated built-ins should expose only non-public internal accessors, such as `dmtcp_Timer_PluginDescr()`, `dmtcp_Sysvipc_PluginDescr()`, `dmtcp_Pid_PluginDescr()`, `dmtcp_IpcSsh_PluginDescr()`, `dmtcp_IpcEvent_PluginDescr()`, `dmtcp_IpcFile_PluginDescr()`, `dmtcp_IpcPty_PluginDescr()`, and `dmtcp_IpcSocket_PluginDescr()`. These declarations belong in internal headers or private namespace declarations consumed by `src/pluginmanager.cpp`, not in the public `include/dmtcp.h` header. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/plugin/timer/timerlist.cpp] [src/plugin/svipc/sysvipc.cpp] [src/plugin/ipc/ipc.cpp] [src/plugin/pid/pid.cpp]

The public ABI remains an `unchanged DMTCP_DECL_PLUGIN` contract. External shared-object plugins may keep using `DMTCP_DECL_PLUGIN`, may keep defining their own `dmtcp_initialize_plugin`, and may keep chaining through `NEXT_FNC(dmtcp_initialize_plugin)` as described by `include/dmtcp.h`. Consolidated internal objects, in contrast, must not keep exported `dmtcp_initialize_plugin` definitions or continue using `DMTCP_DECL_PLUGIN`; their only registration path is the PluginManager-owned built-in table and its internal descriptor accessors. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/plugin/timer/timerlist.cpp] [src/plugin/svipc/sysvipc.cpp] [src/plugin/ipc/ipc.cpp] [src/plugin/pid/pid.cpp]

The accessor body should preserve descriptor data and avoid adding startup side effects. For timer, the tracer bullet can make `timerPlugin` file-local and return it by value from `dmtcp_Timer_PluginDescr()`; SysV IPC and pid should follow the same shape when they move. IPC subplugins should expose by-value accessors for the existing `sshPlugin`, `eventPlugin`, `filePlugin`, `ptyPlugin`, and `socketPlugin` descriptors instead of preserving the old helper functions whose only job was `dmtcp_register_plugin`. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/plugin/timer/timerlist.cpp] [src/plugin/svipc/sysvipc.cpp] [src/plugin/ipc/ipc.cpp] [src/plugin/pid/pid.cpp]

## Consolidated IPC registration design

IPC consolidation should replace the handwritten IPC `dmtcp_initialize_plugin` with table entries that call internal descriptor accessors in the same forward order as today: `ssh,event,file,pty,socket`. The source comments in `src/plugin/ipc/ipc.cpp` explain why the forward order exists: file, pty, and socket are independent relative to each other, event must restore after file-descriptor providers, and SSH must restore after socket because it relies on the restored out-of-band socket. [src/plugin/ipc/ipc.cpp] [src/plugin/ipc/ssh/ssh.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp]

Because `PluginManager::eventHook` processes `DMTCP_EVENT_RESUME` and `DMTCP_EVENT_RESTART` in reverse registration order, the consolidated IPC rows must continue to yield `socket,pty,file,event,ssh` during those restore events. A change that registers socket before SSH in the forward table would appear harmless in the forward listing but would break the intended reverse restart layering. The S03 negative test should therefore assert both the forward marker and the reverse marker, not just the presence of all five descriptors. [src/plugin/ipc/ipc.cpp] [src/plugin/ipc/ssh/ssh.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp]

The IPC implementation migration can keep each subplugin's event hook and singleton state in its current source file, while changing the registration surface from `ipc_initialize_plugin_*()` helper functions to descriptor accessors. The built-in table should contain five explicit IPC rows instead of a nested IPC initializer so order review stays local to `src/pluginmanager.cpp` and so duplicate internal chain nodes cannot reappear through `src/plugin/ipc/ipc.cpp`. [src/plugin/ipc/ipc.cpp] [src/plugin/ipc/ssh/ssh.cpp] [src/plugin/ipc/event/eventconnlist.cpp] [src/plugin/ipc/file/fileconnlist.cpp] [src/plugin/ipc/file/ptyconnlist.cpp] [src/plugin/ipc/socket/socketconnlist.cpp]

## Duplicate dmtcp_initialize_plugin prevention

The consolidation invariant is `no duplicate internal dmtcp_initialize_plugin definitions`: after a built-in moves into libdmtcp, there must be `one libdmtcp dmtcp_initialize_plugin` chain node for DMTCP internals, and there must be `no DMTCP_DECL_PLUGIN for consolidated built-ins`. External shared-object plugins still use the public macro and the public chain; this rule applies only to built-ins that have been folded into libdmtcp. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/plugin/timer/timerlist.cpp] [src/plugin/svipc/sysvipc.cpp] [src/plugin/ipc/ipc.cpp] [src/plugin/pid/pid.cpp]

Implementation failure modes and S03 negative tests should be explicit. First, fail the build or symbol audit if timer, svipc, IPC, or pid still exports its own internal `dmtcp_initialize_plugin` after consolidation. Second, fail the registration-order test if an external `DMTCP_PLUGIN` chain order shift puts built-ins before user descriptors. Third, fail the event-order test if IPC reverse restart order is not `socket,pty,file,event,ssh`, or if pid is not last in the forward built-in descriptor sequence. [src/pluginmanager.cpp] [src/pluginmanager.h] [include/dmtcp.h] [src/plugin/timer/timerlist.cpp] [src/plugin/svipc/sysvipc.cpp] [src/plugin/ipc/ipc.cpp] [src/plugin/pid/pid.cpp]

Fourth, fail alloc/dl validation if disabled wrappers do anything more than fast-pass to real functions or if descriptor rows appear for alloc/dl.
