As we discusseed, if we think of libraries as being layers of software, then ideally, we'd like to think of a function call as being handled first by most wrappers in libraries using RTLD_NEXT, and then libdmtcp.so and friends, and then glibc.so.  So, DMTCP should sit just above libc.so.

This means that the libdmtcp.so constructor should execute very early (just after libc.so).

But it also means that the DMTCP wrappers should be called _last_, just before libc.so.  Because we use LD_PRELOAD=libdmtcp.so, that's not happening.

So, I'd really like to collaborate with you on a roadmap document on how we want to fix that.  (The DMTCP constructor should happen early, which does mostly happen, but the DMTCP wrapers should happen late, which is not happening right now.)

I'd like to test if this scenario would enable us to checkpoint TSAN (thread sanitizer).  The problem here is that libtsan.so has a wrapper for almost everything.  To see this, try:
  nm -D /lib/x86_64-linux-gnu/libtsan.so.2  |grep interceptor

In theory, that doesn't matter, because the TSAN wrappers run very early, _before_ the DMTCP wrappers.

But the TSAN constructor runs very early, before the DMTCP constructor.  So, we depend on the TSAN constructor calling a DMTCP wrapper function, which will then allow DMTCP to run its own constructor.

The issue is that while DMTCP is then running its own constructor, it then calls a function that is wrapped by libtsan.so.  And so, either libtsan.so then crashes, or we get an infinite loop.

And as a concrete example, for our model checker project (McMini/DeepDebugging), we would like to periodically checkpoint.  But checkpointing a target executable compiled with TSAN (thread sanitizer) is a problem.

If you want to experiment, try (where example-tsan.c can just be "hello, world").:
  ${CC} -g3 -fsanitize=thread -pthread example-tsan.c -o example-tsan
  gdb --args bin/dmtcp_launch ./example-tsan

And when you run GDB, do the following:
  break __tsan::Initialize
  break dmtcp_initialize_entry_point
  run
  break dlsym
  continue

Note that since glibc-2.34 merged libdl.so into libdmtcp.so, dlsym is just another libc.so function.  So, the libdl.so functions were merged into glibc.

So, I suppose the puzzle now is that DMTCP is calling dl_iterate_phdr.
And then, TSAN intercepts this first, since DMTCP doesn't have a wrapper for it.

So, I suppose that next, DMTCP could create a wrapper for dl_iterate_phdr,
and then call it first, and force it to call libc:dl_iterate_phdr.
In this way, libtsan has no chance to intercept it.

But this also means that DMTCP must have a wrapper for dl_iterate_phdr that
is first in library search order.

And yet, if I understood your plans, you want to stop using dlsym(), and ensure that DMTCP wrappers would run _late_, just before libc.so.

So, my big question is whether your design will support DMTCP running with TSAN.

For motivation, this is a good use case.  People stress test a multithreaded program by running it for a very long time, to see if TSAN will catch an error.  Long-running programs prefer to be able to use checkpoint/restart.
